# CST — End-to-End Application Design

**Status:** Working design document. Extends `IRC_Street_Designer_Plan_v2.md` (which stays authoritative for stage scope and geometry algorithms) with the decisions that make CST a complete, usable web application: project lifecycle, coordinate math, persistence, data flow, and quality-of-life. Where this document and Plan v2 overlap, Plan v2 wins; where Plan v2 is silent (saving, start flow, UI conventions), this document decides.

---

## 1. The project lifecycle

A CST **project** = one locality, one design. The unit a user opens, works on, saves, exports.

### 1.1 Start flow (location first)

Every project fixes its location before design work — this makes all geometry georeferenced from the first click and keeps the tangent projection accurate (§2).

```
New project
  └─ 1. Find the place: geocode search (Nominatim) or pan/zoom the world basemap
  └─ 2. Fix it: EITHER drag a bounding box → Overpass import inside it
              OR "start blank here" → empty anchored canvas, draw manually
  └─ 3. Work the stages: Network → Sections → Junctions → Detailing → Export
```

- The moment the bbox is confirmed or "start blank" is clicked, the **projection origin** is frozen to the bbox/view center. Everything after is local metres.
- The bbox drag tool replaces today's center+radius fields (keep the fields as the "precise" alternative in a collapsible row). Overpass already takes a bbox; corners come from `toLatLon(view corners)`.
- Manual drawing before any import is always anchored — no "unanchored design" state exists (removes the re-anchoring UI problem entirely).

### 1.2 Additive import (later, explicitly out of MVP)

"Extend my area eastward" = second Overpass call projected into the **existing** origin (`toLocal` already supports this), then a **stitch pass** at the seam: nodes within ε of an existing node merge; ways crossing the seam split against existing edges (the planar-enforcement machinery from drawing already does this). Until stitching exists, import replaces — the UI says so.

## 2. Coordinate math — the settled position

| Concern | Decision | Why |
|---|---|---|
| Working CRS | Local tangent equirectangular: `x = (lon−lon₀)·K·cos(lat₀)`, `y = −(lat−lat₀)·K`, K=111320 | Exact enough (< 10 cm error within 2 km of origin), trivial inverse, metres everywhere |
| Origin | One per project, frozen at start, stored with the graph, undoable | Single link between design and world; DXF/GeoJSON georeferencing needs exactly this |
| Re-anchoring | Explicit operation only: re-project every coordinate through old→new origin. Never implicit. | Silent origin changes desync data from basemap |
| Basemap interplay | MapLibre camera derives from view center via inverse projection; zoom from `px/m` at center latitude | Sub-pixel agreement at project scale; divergence only matters >10 km from origin, which §1.1 prevents |
| Distortion budget | Projects are ≤ ~4 km across. Beyond that, warn on import ("area too large for a street-design project") | A street design project is inherently local; enforcing it keeps every simplification valid |

**Geometry robustness roadmap** (in priority order, all prefigured in Case_Study):
1. **Clipper2 (WASM) as safety net** — union/clean output polygons before export; kills self-intersecting tapers on sharp bends (known limitation of the miter/sample method).
2. **Probe-method junction boundary** (LaneMaker) for shallow-angle/5+ approaches — fixes the corner-spike on near-parallel approaches (visible today at the flyover).
3. **Cubic-Bézier corner fitting** for junction corners (ConnectRays case 2) — designed curb radii instead of straight corner cuts.
4. **Ported osm2streets `PolyLine`** as the one blessed primitive library once tests exist — today's ad-hoc helpers migrate into it.

## 3. Data model: stored vs derived (the dirty-flag contract)

The rule from Plan v2 §1.2 is the architecture: **stored state is minimal and parametric; everything visual is derived and regenerates freely; user-touched derived artifacts get a STALE badge instead of silent regeneration.**

| Data | Status | Notes |
|---|---|---|
| nodes, edges (geometry, tags) | **stored** | the graph core; undoable |
| edge.section (materialized components) | **stored** | copy-on-write from catalog, provenance via catalogId |
| edge.overrides (station-anchored) | **stored** | data model kept; UI is the split-node workflow |
| projection origin | **stored** | with the graph, undoable |
| ribbons, markings | derived | always regenerate |
| node transitions | derived | always regenerate (fully parametric) |
| junction polygons | derived **until edited** | when junction editing lands (types, radii, elements), a hand-touched junction stores its parameters + `autoGenerated: false`; upstream edits mark it STALE with regenerate/keep |
| furniture (Stage 3) | stored parametrically | `{edgeId, station, component, side}` anchors; rule provenance (`placedBy`) |
| movements, turn classes | derived | Phase 2.5 |
| validation issues, review list | derived / session | never persisted |

STALE UX (when it arrives): badge on the artifact + one batched prompt per edit gesture, defaulting to regenerate for untouched things (Plan v2 §9.1).

## 4. Persistence — save, retrieve, share

### 4.1 Format: one JSON document

```jsonc
{
  "format": "cst-project",
  "version": 1,                    // integer, bumped on breaking change
  "meta": { "name": "JM Road redesign", "createdAt": ..., "modifiedAt": ... },
  "origin": { "lat": 18.5289, "lon": 73.8478 },
  "graph": { "nodes": {...}, "edges": {...}, "nextNodeNum": ..., "nextEdgeNum": ... },
  "junctionEdits": {},             // hand-touched junction params (later)
  "furniture": [],                 // Stage 3 (later)
  "view": { "x": ..., "y": ..., "scale": ..., "basemap": "osm" }   // convenience, non-essential
}
```
Serialization is exactly the undo partialize slice + meta — one function, no second schema. Derived artifacts are never saved. **Migration policy:** loader switches on `version`, upgrades stepwise (v1→v2→…), refuses unknown majors with a clear message. Old files must always open.

### 4.2 Where it lives

1. **Autosave → localStorage** (MVP, cheap): debounced (~2 s) write of the current project under `cst:autosave`; on boot, offer "Resume last session / Start fresh". Survives refreshes and crashes — the single biggest QoL gap today.
2. **Save/Open as file** (MVP): `Download .cst.json` / file-picker open. Serverless persistence + shareability + backups, consistent with the client-only lock-in.
3. **IndexedDB project list** (post-MVP): named projects, thumbnails (canvas snapshot), modifiedAt — a "recent projects" start screen. localStorage stays the crash-recovery layer.
4. **Cloud/collab**: out of scope by architecture (client-only). The file format is the interchange; if a backend ever appears it syncs these documents.

### 4.3 What "New project" means

Clears graph + origin + review/session state, keeps catalog/settings, routes to the start flow (§1.1). Guard: if unsaved changes exist (dirty since last export/autosave differs from last explicit save), confirm.

## 5. Data flow (how state moves)

```
user input (Konva) ──► store actions ──► graph slice (zustand, zundo history)
                                             │
              useMemo derivations (per render, pure functions of graph):
                ├─ edge geometry: spans → ribbons + markings (+ trims applied)
                ├─ node artifacts: junction polys, node transitions, per-end trims
                ├─ validation issues
                └─ (later) movements, furniture placements, compliance report
                                             │
                                   Konva layers render
basemap: view state ──► MapLibre camera (one-way; MapLibre never owns input)
persistence: graph slice ──► debounced serializer ──► localStorage / file
```

Principles that must hold as it grows:
- **One store, one history.** Stages are lenses, not databases. Selection/tool/UI state stays out of the undo partialize.
- **Derivations are pure functions of the graph slice** — that's what makes save files small, undo trivial, and regeneration always safe.
- **Derivation cost scaling:** today, whole-graph derivation per change is fine (~100 edges). At ~1000+ edges: memoize per-edge/per-node keyed by identity (structural sharing already supports this), then move `deriveNodeArtifacts` to a worker if profiling demands. Don't do either early.

## 6. UI & quality of life (the gap between tool and application)

**Global**
- Start screen (§1.1) instead of dropping into an empty canvas.
- Autosave indicator ("saved ✓ / saving…") in the header; Save/Open/New in a small project menu.
- Command palette-lite: `?` shows the keyboard map. Current keys (Esc/Enter/Del/Ctrl-Z/Y, Shift-angle) + `1/2/3/4` for stages, `V/D/X` select/draw/split, `F` fit-to-network.
- Consistent selection model: click selects, Esc walks back (draft → tool → selection), panel lists and canvas always agree (done today; keep it law).

**Canvas**
- Scale bar (we have px/m; draw the bar) and a north arrow once rotation exists (don't add rotation before export needs it).
- Measure tool (M): click-click distance readout — constant request in design tools, trivial with existing snapping.
- Hover tooltips: street name/class/section on hover after 400 ms.
- Basemap opacity slider (satellite at 100 % fights the schematic; 60 % default when designing).

**Stage-specific**
- Network: lasso/marquee select for bulk delete of alleys; "delete all service roads" utility.
- Sections: the review list is the pattern — extend it to compliance warnings (below-minimum components across the whole network in one list, click-to-focus).
- Junctions: focused junction view (Plan §4.3) — selecting a junction dims everything beyond ~40 m and shows movement arrows once Phase 2.5 lands.
- Detailing (future): every rule-placed item click-traces to its rule + citation ("placed per IRC:103 §x").

**Trust surface** — the app must always answer "why does it look like this": validation panel (exists), review list (exists), STALE badges (future), rule provenance (future). These four are the same idea: no silent magic.

## 7. Export (Stage 4) — the deliverable defines the app

The user's output is a **plan someone can act on**: print it, open it in CAD, overlay it in GIS.

- **SVG/PDF sheet composition:** plan at a chosen scale (1:500/1:1000) + legend (auto from used kinds — exists), scale bar, north arrow, title block, notes incl. rule citations and "(verify)" flags. Sheet templates (A1/A3) as JSON config. SVG is the master; jsPDF+svg2pdf renders it.
- **DXF:** one layer per component kind + `CENTERLINE`, `JUNCTION`, `MARKING`, `ROW_EDGE`, `FURNITURE`; closed LWPOLYLINEs; `$INSUNITS=6` (metres); origin lat/lon written to a `NOTES` layer for georeferencing. The derived-polygon model maps 1:1 — this is mapping work, not research.
- **GeoJSON:** every derived polygon + centerlines through `toLatLon(origin, …)` — instant QGIS overlay. Cheapest export and the best debugging tool; build it first.
- Component polygons must pass through Clipper2 clean-up before export (§2) so CAD users never receive self-intersecting rings.

## 8. Quality practice (how we keep it correct)

- **Geometry fixtures:** the ported algorithms (offset, junction corners, transitions, DC merge) get unit tests with known inputs — osm2streets ships test fixtures worth borrowing (Case_Study §8.2). This is the highest-value testing money in the project; UI churn is cheap, geometry regressions are silent.
- **The Playwright drives are the regression suite:** each phase's done-when script (import → clean → merge → draw → sections → junctions) already exists from development; keep them runnable (`app/tests/`) and run before deploy.
- **Perf budget:** interaction at 60 fps with 300 sectioned edges; derivation < 50 ms per graph edit at that size. Measure before optimizing (§5).
