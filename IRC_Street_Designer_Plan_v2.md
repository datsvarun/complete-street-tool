# IRC Street Designer — Implementation Plan v2 (Stage-Based Workflow)

**Status:** Supersedes `IRC_Street_Designer_Plan.md` where they conflict. Everything not mentioned here (stack, case-study findings, junction geometry algorithms, domain model core) carries over unchanged from v1 + `Case_Study.md`.

**What changed vs v1:** The app is now organized as an explicit **5-stage workflow** (Graph → Sections → Junctions → Fine-tuning → Export, with 3D as a later toggle). Four genuinely new capabilities appear: (A) a first-class **graph editor** with drawing + cleaning, (B) **smooth priority-based section transitions**, (C) a **rule-based furniture/detailing engine**, (D) **DXF export**. This document specifies those four in depth and restructures the phases around the stages.

---

## 0. Assumptions & pushback (read first)

1. **OSMnx is out.** OSMnx is a Python library — it cannot run in a client-only browser app. The equivalent capability (download OSM ways → build a node-edge graph → simplify) is achieved with **Overpass API + a TypeScript graph builder** (already specced in v1 Phase 1 / Case_Study §1). If you later want batch/city-scale preprocessing, OSMnx can live in an *offline* Python script that emits GeoJSON the app imports — but not in the app itself.
   *Assumption: client-only remains a locked decision. If not, say so and OSMnx becomes viable server-side.*

2. **Stages are a workflow, not a straitjacket.** You described "save and move to the next step." I recommend **soft stage-gating**: the UI presents stages 1→4 as tabs with a completion checklist per stage, but the domain model is one shared store and the user can go back at any time. Hard gating (freeze the graph after Stage 1) breaks your own requirement that users can add nodes mid-design to change widths. The rule instead is: **every downstream artifact is derivable/regenerable from upstream state** (see §1.2, the dirty-flag system).

3. **Dual carriageways as single edges: agreed, with a caveat.** Representing a divided road as one edge with `{carriagewayType: 'divided', medianWidth}` is the right model *for design*. But OSM maps them as two parallel ways. osm2streets' own `MergeDualCarriageways` is WIP/broken (Case_Study §1.3). So for MVP: **detection + assisted manual merge** (tool highlights parallel one-way pairs, user clicks "merge", we synthesize a centerline midway) — not automatic merging. Automatic merging is a research problem; don't block the app on it.

4. **Flyovers are attributes, not geometry, in 2D.** `edge.attributes.flyover = { present: true, span: [startM, endM], type: 'vehicular' }`. Rendered as a hatched overlay band in 2D. The at-grade street underneath is what gets the IRC section. Junctions under flyovers get `junctionType: 'grade-separated'` which mainly changes which junction elements are offered (no signals on the flyover span, crossings under the piers, etc.). Full flyover geometry is a 3D/post-MVP concern.

5. **"As good as an urban design plan" is a rendering + rules problem, not more geometry.** The Kumar Park-quality output comes from Stage 3's detailing rules + a good schematic style config, both of which are data, not code. Budget accordingly: the geometry engine is ~60% of the work, but the last 40% (rules + styling) is what makes the output credible.

## 1. Architecture: stages over a single domain model

### 1.1 The five stages, mapped to app modes

| Stage | Mode name | Primary state touched | Output |
|---|---|---|---|
| 1 | **Network** | `graph` (nodes + edges) | Clean geospatial node-edge graph |
| 2 | **Sections** | `edge.crossSection`, `edge.sectionOverrides`, `transitions` | Streets as ribbons |
| 2A | **Junctions** | `junction.*` | Junction polygons + elements |
| 3 | **Detailing** | `furniture[]`, `crossings[]` | Placed elements w/ rule provenance |
| 4 | **Export** | none (read-only) | PDF / GeoJSON / DXF / SVG |
| 5 | **3D** (post-MVP) | none (derived view) | R3F scene |

One Zustand store, one undo history (Zundo), stage = a UI mode that filters which tools/layers are active.

### 1.2 Derivation graph + dirty flags (the key architectural addition)

Because users edit upstream state (move a node) after downstream work exists (a hand-tuned junction), we need explicit dependency tracking:

```
graph (nodes, edges)
  └─→ edge ribbons        (recompute: cheap, always auto)
  └─→ junction polygons   (recompute: auto IF junction.autoGenerated, else mark STALE)
        └─→ junction elements (crossings, islands: re-anchor by parameters, else STALE)
  └─→ transitions          (recompute: always auto — they're fully parametric)
  └─→ furniture            (re-run placement rules on affected edges only; user-pinned items re-anchor or STALE)
```

Rule: **auto-generated things silently regenerate; user-touched things get a STALE badge and a "regenerate / keep" prompt.** This is the single mechanism that makes the whole "dynamic, edit-anything" requirement tractable. Every downstream object stores its anchor *parametrically* (e.g., crossing at `{edgeId, distanceFromJunction: 8m}`, not absolute coords), so re-anchoring after upstream edits is usually automatic.

### 1.3 Domain model extensions (delta over v1 §3)

```ts
// Node classification is DERIVED, never stored:
type NodeClass = 'terminus' /*deg 1*/ | 'bend' /*deg 2*/ | 'junction' /*deg ≥3*/;

Edge (was Street)
  ├── ... (all v1 Street fields)
  ├── attributes: {
  │     carriagewayType: 'undivided' | 'divided',
  │     medianWidth?: number,
  │     flyover?: { span: [number, number], clearance?: number },
  │     oneway: boolean,
  │   }
  └── (sectionOverrides now generate Transition objects at their boundaries)

Transition (NEW — always derived, never hand-edited in MVP)
  ├── edgeId
  ├── atM              // centerline station where sections change
  ├── fromSection / toSection   (refs)
  ├── length           // computed from taper rule, see §3.2
  └── perComponentTaper: { component, side, action: 'taper'|'drop'|'hold' }[]

Junction
  ├── ... (v1 fields)
  ├── junctionType: 'general' | 'signalized' | 'roundabout' | 'grade-separated'
  ├── template?: JunctionTemplateRef   // pre-seeds elements per type, see §4.2
  └── movements: Movement[]            // from Phase 2.5 turn classification

FurnitureItem (NEW)
  ├── kind: 'busStop' | 'bin' | 'bench' | 'streetlight' | 'tree' | 'curbCut' | 'bollard' | ...
  ├── anchor: { edgeId, station: number, component: string, side: 'left'|'right' }  // parametric!
  ├── placedBy: 'rule:<ruleId>' | 'user'
  └── params: Record<string, number>
```

## 2. Stage 1 — Network editor (graph build, draw, clean)

v1 Phase 1 covered OSM→graph. New here: **drawing** and **cleaning tools**.

### 2.1 Import path (unchanged from v1, summarized)
Overpass query for `highway=*` in bbox → parse ways/nodes → split ways at shared nodes → graph. Port osm2streets transforms: `CollapseShortRoads`, `CollapseDegenerateIntersections` (run twice, per Case_Study §1.3). Project to local metric CRS immediately.

### 2.2 Draw path (NEW)
- Tool: click to place vertices, double-click/Esc to end an edge. Implemented on a **Konva canvas overlay** (not mapbox-gl-draw — we need snapping and custom handles; mapbox-gl-draw is fine for the POC but won't carry the editor).
- **Snapping** (the whole game here): snap-to-node (r = 8px screen-space), snap-to-edge (projects onto nearest edge; committing splits that edge and inserts a node), snap-to-angle (hold Shift: 15° increments), snap-to-extension of the last segment.
- Drawing across an existing edge auto-nodes the crossing (planar enforcement) — with an opt-out modifier key for the flyover case where lines cross without connecting.

### 2.3 Cleaning toolkit (NEW)
Each is a small tool + the transform pipeline:
- **Merge nodes** (drag one onto another, or marquee-select + "merge") → rewires edges, dedupes.
- **Split edge** (click on edge → insert node). This is also how mid-street section changes get anchored later — but note sectionOverrides usually make splitting unnecessary; offer both.
- **Delete edge/node** with orphan cleanup.
- **Simplify** (Douglas-Peucker on bend chains, tolerance slider) — collapses stair-step OSM geometry.
- **Assisted dual-carriageway merge** (per §0.3): detector = pairs of edges that are (a) both oneway, (b) opposite bearings within 20°, (c) mutual Hausdorff distance < 25m, (d) share both end junctions or nearly so. Highlight candidates; on confirm, synthesize midline (average of the two polylines after arc-length reparameterization), set `carriagewayType: 'divided'`, `medianWidth = mean separation − carriageway halves`.
- **Node classification badges** rendered live (terminus/bend/junction by degree) — derived, zero storage.

**Done when:** import a messy 500m Pune extract, merge one dual carriageway pair, draw one new street that snaps into the network, delete a service alley, and the graph validates (no dangling refs, no zero-length edges, planar except flagged flyover crossings).

## 3. Stage 2 — Sections & transitions

v1 Phase 2 covers catalog picking, the Streetmix-style strip editor, offset rendering, and overrides. New here: the **transition engine** — v1 punted with "hard cut for MVP." Your requirement upgrades it to a core feature. Correctly so: transitions are what make mid-street width changes look designed instead of broken.

### 3.1 Component priority table (data, not code)

When ROW shrinks, components give way in a defined order. Encode as ranked, citable data alongside the IRC catalog:

```json
{
  "reductionPriority": [
    { "component": "On-street Parking",  "action": "drop",         "rank": 1 },
    { "component": "Plantation/Buffer",  "action": "shrinkTo", "min": 0.5, "rank": 2 },
    { "component": "MUZ",                "action": "shrinkTo", "min": 1.0, "rank": 3 },
    { "component": "Median",             "action": "shrinkTo", "min": 0.6, "rank": 4 },
    { "component": "Footpath",           "action": "shrinkTo", "min": 1.8, "rank": 5 },
    { "component": "Cycle Track",        "action": "shrinkTo", "min": 2.0, "rank": 6 },
    { "component": "Carriageway",        "action": "hold",                  "rank": 99 }
  ]
}
```

Minimums come from the IRC catalog/SP:118 + IRC:103; every entry cites source (Streetmix citable-data pattern, Case_Study §2.2). This table also powers a bonus feature: **"fit section to ROW"** — given a target width and a template, auto-shrink/drop by priority to produce a compliant narrower variant.

### 3.2 Transition solver

Given adjacent sections S₁ (e.g., 24m) and S₂ (21m) on one edge:

1. **Component matching:** align S₁ and S₂ components by (element type, side) using ordered matching from each curb inward. Unmatched in S₂ → `drop`; unmatched in S₁ → `introduce`. Same type, different width → `taper`.
2. **Transition length:** per component, `L = Δwidth × taperRatio(component)`. Carriageway edges: 1:10 urban (IRC-consistent for ≤50 km/h; expose as setting). Footpath/landscape: 1:3–1:5 (visual, not vehicular). Transition zone length = max over components, clamped to [3m, available edge length/2].
3. **Geometry:** within the zone, each component boundary offset is interpolated `offset(t) = lerp(o₁, o₂, smoothstep(t))`. Smoothstep (3t²−2t³) gives tangent-continuous entry/exit — the "smooth in and out angles" you asked for — with no arc-fitting needed. Dropped components taper to zero width then their polygon simply ends.
4. **Alignment rule:** carriageway centerline holds position by default (your 24→21→24 case: carriageway continuous, flanks flex). Option per transition: `align: 'center' | 'leftCurb' | 'rightCurb'` for the common Indian case where one property line is fixed.
5. Output is a `Transition` object; the ribbon renderer treats the zone as just another set of component polygons. Fully derived → always safe to regenerate.

**Done when:** apply 24m, add a 21m override mid-edge, see parking drop and MUZ shrink through two smooth ~10–15m tapers, then restore — and dragging the override handles live-updates the tapers.

### 3.3 Auto-section assignment on entry to Stage 2 (NEW)
On first entering Sections mode: for each edge, map `highway=*` → IRC ROW class (defaults table), check `width=`/`lanes=` tags if present, assign the best-matching catalog section, and queue a **review list** of edges where confidence is low (no tags, or tag-derived width far from any catalog ROW). The prompt-the-user flow you described = this review list, not modal interruptions.

## 4. Stage 2A — Junction editor

v1 Phase 3 + Case_Study two-tier boundary strategy stand as written (4-case dispatch default; LaneMaker probe method for ≥5 approaches / asymmetric widths). New here: **junction types as templates**.

### 4.1 Junction type assignment
Per node (degree ≥3): `general | signalized | roundabout | grade-separated`. Auto-seed from OSM tags where present (`highway=traffic_signals` on the node → signalized; `junction=roundabout` ways → roundabout). Type is editable.

### 4.2 Type = template = pre-seeded parametric elements
A template is a function `(junction, movements) → JunctionElement[]`:

- **General:** corner fillets (radius by street class), zebra crossings on approaches with footpaths, refuges where crossing length > 12m (IRC:103 threshold — verify against your copy).
- **Signalized:** general + stop lines at `crossingOffset + 1m`, signal head markers per approach, optional dedicated turn pockets (generated as sectionOverrides on approach edges — reuses the §3 machinery, no new geometry).
- **Roundabout:** central island (radius parametric, seeded from inscribed-circle fit of the junction polygon), circulatory carriageway width from largest approach, splitter islands on every approach (teardrop generator from v1 3b), deflection check as a warning.
- **Grade-separated:** at-grade elements only under the flyover span; piers as furniture markers.

Applying a template populates `elements[]` — everything remains individually editable afterward with the v1 3b tools (corner radius, splitter, refuge, crossing, channelized turn). Re-applying a template = regenerate (with STALE protection per §1.2). This gets "different junction design for different angles, shapes and features" without a combinatorial UI: templates handle 80%, primitives handle the rest.

### 4.3 Focused junction view
Selecting a junction opens a zoomed edit mode: approaches clipped to ~40m, movement arrows shown (Phase 2.5 turn classification), element toolbar active, everything else dimmed. This is a camera + layer-filter change, not a separate document.

## 5. Stage 3 — Detailing engine (rule-based furniture & crossings)

Entirely new. This is a **forward-chaining rule pass over the finished network**, emitting `FurnitureItem`s with provenance.

### 5.1 Rule format — data-driven, citable

```json
{
  "id": "midblock-crossing-spacing",
  "source": "IRC:103-2012 §x.x",
  "appliesTo": { "edgeClass": ["arterial", "sub-arterial"], "minLengthM": 250 },
  "generate": {
    "kind": "crossing", "type": "raised",
    "spacing": { "maxM": 250, "preferNear": ["busStop", "junction"] }
  }
}
```

Initial rule set (each cites IRC:103 / SP:118 / ITDP Better Streets / NACTO GDCI — numbers below are placeholders to verify against the documents):
1. **Mid-block crossings:** max spacing 200–250m on arterials; snap to desire points (bus stops, edge midpoints).
2. **Bus stops:** on `busRoute`-tagged or arterial edges, spacing 300–400m, placed **far-side of junctions** ≥20m past the crossing, on the footpath/MUZ component; generates a bulge sectionOverride if a bus bay is chosen.
3. **Curb cuts / ramps:** at every crossing end and every junction corner with a footpath — purely derived from crossings, zero user input.
4. **Street trees:** in Plantation/MUZ components, spacing 8–12m, suppressed within 10m of crossings and junction corners (sight lines).
5. **Streetlights:** alternating/opposite per carriageway width, spacing ≈ 25–30m (staggered if ROW > 18m).
6. **Bins/benches:** near bus stops and crossings, on MUZ, min clear footpath maintained (checks §3.1 minimums).
7. **Bollards:** across footpath continuations at junction mouths on local streets.

Execution: rules run in priority order; each sees previously placed items (spacing rules need this); conflicts resolved by clear-width check on the host component. Every placed item is draggable-along-station and deletable; user edits set `placedBy: 'user'` and survive re-runs.

### 5.2 Why this design wins
- Rules are JSON → adding a guideline (Tender SURE, a municipal DPR standard) is data entry, not code.
- Provenance per item → the export legend can print "crossings per IRC:103-2012" — this is what makes the output *defensible* as a code-compliant plan.
- Parametric anchors (§1.3) → furniture survives upstream edits.

**Prerequisite task:** extract the actual numeric standards from IRC:103, IRC:35 (markings), SP:118 into a `rules.json` with citations. This is domain work you're better placed to do than any library; budget 2–3 focused days with the documents.

## 6. Stage 4 — Export

v1 Phase 5 (SVG/PNG/GeoJSON/JSON/IndexedDB) stands. New: **DXF**.

### 6.1 DXF
- Library: **`@tarikjabiri/dxf`** (pure TS, browser-safe, actively maintained; writes polylines, hatches, layers, text). Fallback: `dxf-writer`.
- Mapping: one DXF **layer per component type** (`CARRIAGEWAY`, `FOOTPATH`, `CYCLE_TRACK`, `MEDIAN`, `CROSSING`, `FURNITURE`, `CENTERLINE`, `ROW_EDGE`...), polygons as closed LWPOLYLINEs (+ optional HATCH), furniture as blocks/point+text, all in the **local metric CRS** with the projection origin written to a `NOTES` layer so a CAD user can georeference. Units: meters, `$INSUNITS = 6`.
- This is genuinely a few days of mapping work, not a hard problem — the domain model is already "polygons with types."

### 6.2 PDF
SVG compose (plan + legend + scale bar + north arrow + rule-provenance notes) → jsPDF via svg2pdf. Sheet templates (A1/A3, title block) as a JSON config.

## 7. Stage 5 — 3D (post-MVP, design for it now)

Keep the v1/Case_Study position: don't build 3D in MVP. Do two cheap things now so it stays cheap later:
1. **Quantized elevation on components** (integer curb levels × 0.15m — 3DStreet's pattern, Case_Study §3.3). Add the field to `Component.meta` from day one.
2. Keep all geometry output as typed polygons + centerline stations — the R3F view is then extrusion + asset placement at furniture anchors. Toggle view (per your earlier decision), Zustand as shared state.

## 9. Risks (delta over v1)

1. **Dirty-flag UX** — regenerate-vs-keep prompts can get annoying. Mitigate: batch prompts per edit gesture; default to regenerate for anything untouched.
2. **Transition solver on curved centerlines** — offsets interpolating across a bend can self-intersect. Mitigate: clip transitions away from high-curvature spans; Clipper2 union as the safety net (already in stack).
3. **Rules.json content risk** — wrong numbers = wrong plans with false authority. Mitigate: every rule cites clause; validator marks "verify" until you've checked the source; this is your domain-expert task, not the code's.
4. **Scope creep in Stage 1** — a graph editor can absorb infinite polish. Mitigate: the done-when list in §2.3 is the whole scope; anything else waits.
